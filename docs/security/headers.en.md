# Security Headers & CSP

> **[Русская версия](headers.md)**

## Overview

Security headers are applied at the proxy layer. Every request goes through `proxy.ts → handleProxy(...)` and is wrapped with `applySecurityHeaders(response, isStaticFile, request)` from `lib/security/headers.ts`. Two header sets exist depending on the resource:

- **Static files** (under `/_next/static`, etc.) — minimal CORS only. CSP is intentionally not applied to avoid HTTP/2 protocol errors that some browsers raise when CSP collides with content-type sniffing on `.svg`/`.woff2`.
- **Everything else** — full CSP, HSTS (in production), `X-XSS-Protection`.

Note: there is no top-level Next.js `middleware.ts`. The proxy lives in `proxy.ts` at the repo root and is wired up via the Next.js Edge runtime config.

## Content-Security-Policy

Generated by `generateCSPHeader(isDev)` in `lib/security/headers.ts`. Header is set on every non-static response:

```
default-src 'self' rvn.market *.rvn.market;
script-src 'self' 'unsafe-eval' 'unsafe-inline' rvn.market *.rvn.market https://challenges.cloudflare.com;
style-src 'self' 'unsafe-inline' rvn.market *.rvn.market;
img-src 'self' blob: data: rvn.market *.rvn.market *;
font-src 'self' rvn.market *.rvn.market;
connect-src 'self' https://challenges.cloudflare.com rvn.market *.rvn.market *;
frame-src 'self' https://challenges.cloudflare.com rvn.market *.rvn.market;
child-src 'self' https://challenges.cloudflare.com rvn.market *.rvn.market;
object-src 'none';
base-uri 'self';
form-action 'self';
frame-ancestors 'none';
upgrade-insecure-requests;        // production only
```

In development, the directives also include `localhost:* http://localhost:* ws://localhost:* wss://localhost:*` and `upgrade-insecure-requests` is dropped so HMR works on plain HTTP.

### Directive-by-directive rationale

| Directive | Value | Why |
|-----------|-------|-----|
| `default-src` | `'self' rvn.market *.rvn.market` | Restrict everything not otherwise specified to the registrable domain. |
| `script-src` | `'self' 'unsafe-eval' 'unsafe-inline' …` + Cloudflare Turnstile | **Known weak point.** `'unsafe-eval'` is required by some current code paths (Next.js dev runtime, Recharts, Drizzle SQL templates rendered in client bundles). `'unsafe-inline'` is required while inline `<style>`/`<script>` injections from third-party libs (e.g. Turnstile bootstrap) remain. Should migrate to nonces or hashes — tracked as a follow-up. |
| `style-src` | `'self' 'unsafe-inline' …` | `'unsafe-inline'` covers Tailwind's preflight inlining, GSAP-injected styles, and `next/font` runtime styles. |
| `img-src` | `'self' blob: data: rvn.market *.rvn.market *` | `*` allows fetching avatars from any external OAuth provider (Google, Yandex, Twitch, VK avatar URLs). Without `*` first-login avatars wouldn't render. `data:` is for inline placeholder thumbhashes. |
| `connect-src` | `'self' Turnstile rvn.market *.rvn.market *` | Tightening this is one of the items in the security backlog. The wildcard exists because some code makes outbound `fetch` to provider endpoints from the browser (e.g. profile-picture proxying). |
| `frame-src` / `child-src` | `'self' Turnstile …` | Cloudflare Turnstile renders inside an iframe. Embedding our own subdomains is allowed for iframe-based flows. |
| `object-src` | `'none'` | No legacy plugin objects. |
| `frame-ancestors` | `'none'` | We are never embedded; equivalent to `X-Frame-Options: DENY` (we don't set the legacy header). |
| `upgrade-insecure-requests` | production only | Force HTTP→HTTPS on any accidental `http://` URL the bundle still ships. |

The "weak" directives (`'unsafe-eval'`, `'unsafe-inline'`, `connect-src *`, `img-src *`) are documented liabilities. They are tracked in `rvn-web-review.md` and the overall security backlog. The intent of CSP today is defense in depth + frame-ancestors enforcement, not full XSS mitigation.

## HSTS

Production only:

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

`max-age = 1 year`. `includeSubDomains` covers `*.rvn.market`. `preload` flags us for inclusion in the browser preload list (which we have not actually submitted yet — submitting requires an audit run on hstspreload.org).

Not set in development so plain `http://localhost:3000` works. Not set on static files because of HTTP/2 sensitivity (see below).

## Other Headers

- `X-XSS-Protection: 1; mode=block` — legacy; kept for older browser support, no-op in modern Chromium/WebKit.
- `Access-Control-Allow-Origin` — controlled by `lib/security/cors.ts`. In production, restricted to `ALLOWED_ORIGINS` env var; in development, reflects the request origin. Static files allow same-origin and known subdomains via `isValidOrigin()` / `isSubdomain()`.

There is no `X-Frame-Options` header — `frame-ancestors 'none'` in CSP is the modern equivalent and is what gets enforced. There is no `X-Content-Type-Options: nosniff` set globally either; the project relies on Next.js' built-in content type for routes plus the `Content-Type: image/svg+xml` override for static `.svg` (set in `applySecurityHeaders` only when not already present).

## Static Files & HTTP/2

Static-file responses (`isStaticFile = true`) skip security headers entirely:

```ts
if (isStaticFile) {
  // CORS only
  // Optional content-type fix-up for .svg
  return; // no CSP, no HSTS
}
```

This is a workaround for HTTP/2 protocol errors that surface when a CSP header collides with content-type sniffing on certain CDN configurations. The `_next/static` payload doesn't execute in a security context anyway (it's already content-addressed by Next.js — files have hashes in their names), so dropping CSP there is acceptable.

## Origin Validation

`isValidOrigin(origin)` is intentionally permissive:

```ts
export function isValidOrigin(origin: string): boolean {
  return origin.includes(MAIN_DOMAIN);
}
```

It uses substring matching on `MAIN_DOMAIN`. This is **a known weakness** documented in the security review — `https://evil-rvn.market.attacker.com` would match. The plan is to switch to `URL.hostname.endsWith('.' + MAIN_DOMAIN) || hostname === MAIN_DOMAIN`. For now, defense relies on the upstream proxy (Cloudflare) restricting traffic to the canonical domains.

## Configuration

Most of the headers are not configurable at runtime. The only env-driven knobs:

| Variable | Used by | Effect |
|----------|---------|--------|
| `ALLOWED_ORIGINS` | `lib/security/cors.ts` | Comma-separated list of allowed CORS origins in production. |
| `NEXT_PUBLIC_DOMAIN` | `appConfig.domains.main` | The "main domain" used in CSP `default-src` / `*.<domain>`. Falls back to `'rvn.market'` if unset. |
| `NODE_ENV` | `applySecurityHeaders` | Toggles HSTS and the localhost CSP exemptions. |

Changing the CSP itself requires editing `generateCSPHeader()` in code.

## Files

- `lib/security/headers.ts` — `generateCSPHeader`, `applySecurityHeaders`, `isValidOrigin`, `isSubdomain`.
- `lib/security/cors.ts` — `setCorsHeaders`, `handleCorsPreflight`.
- `proxy.ts` — proxy entry point that wraps every response.
- `lib/proxy/auth.ts`, `lib/proxy/protection.ts` — also call `applySecurityHeaders` on synthetic responses (rate-limit denials, suspicion blocks).
- `lib/utils/config.ts` → `appConfig.domains` — main / panel / api domain config.
